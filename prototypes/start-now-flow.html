Design prototype · BAAM Review

Full Service · “Start Now” flow

Customer pays first, BAAM team handles the rest. Six stages: pricing CTA → Stripe Checkout (with business info fields) → welcome page → auto-confirm email → team notification → staff Onboarding queue. All locations belong to one “BAAM Operations” account; customer_records table bridges payment to GBP connection.

Data model — the two new pieces

EXISTING TABLE — one new FK

locations

... existing columns ...
customer_record_id (FK → customer_records, nullable)

Locations from regular signup have customer_record_id = null. Locations from Start Now have it set, providing the trace back to the original Stripe payment.

No changes to users, accounts, RLS policies, or location_subscriptions. The “is this location paid?” check stays as it is today: a row in location_subscriptions linking the location to a Stripe subscription means paid.

Implementation scope & open questions

Backend (1 day)

  • Migration 0029: customer_records table + locations.customer_record_id FK
  • POST /api/billing/start-fullservice: creates Stripe Checkout session with custom_fields and 30-day trial
  • Webhook handler for checkout.session.completed: insert customer_records row, send 2 emails
  • Server action connectCustomerGbp(customer_record_id, place_id): creates location + location_subscriptions row

Frontend (1 day)

  • “Start now” button on pricing page (en + zh)
  • /start/welcome page (Stage 3 above)
  • /app/onboarding queue page (Stage 6 above)
  • GBP picker modal updated to handle the “onboarding queue” entry point (Stage 7)
  • Billing-status pills on the locations list (Stage 8)

Open questions before we build

  • Manager email address: is manager@baamplatform.com the right address for customers to invite? Or do you have an existing operations inbox?
  • Magic-link login for the customer: the welcome email offers “Log in to my dashboard.” Should we build a separate read-only customer-facing dashboard, or is that for v2?
  • Stripe custom_fields: Stripe Checkout supports up to 2 custom fields. Business name + business address fits. If we ever need a 3rd field (e.g., language preference), we’d need a separate post-Checkout form.
  • Day-5 / Day-7 alerts: daily cron with Slack/email alerts for stalled onboarding. Build now or after first 10 customers?
